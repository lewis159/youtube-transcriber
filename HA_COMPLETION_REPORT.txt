WHAT WAS BUILT
================================================================================

1. NGINX LOAD BALANCER (Port 4000 - Public Entry Point)
   - Reverse proxy with round-robin load balancing
   - Automatic failover to healthy instances
   - WebSocket and long-request support
   - Logs all traffic for monitoring

2. TWO APP INSTANCES (Ports 4001-4002 - Internal)
   - Identical configurations running in parallel
   - Share Redis for session management
   - Monitored by Docker health checks every 30s
   - Auto-restarted if unhealthy

3. SHARED REDIS (Port 6379 - Internal)
   - Session storage for both app instances
   - Accessible only from app instances

4. DOCKER HEALTH CHECKS
   - Every 30 seconds, queries /api/health endpoint
   - 10-second timeout with 3 retries
   - Automatic container restart on failure
   - nginx observes health and routes traffic

5. GRACEFUL SHUTDOWN SYSTEM
   - SIGTERM signal handler in health endpoint
   - Stops accepting new requests during shutdown
   - Allows existing requests to complete (30s max)
   - Enables zero-downtime rolling updates

6. REBOOT TRACKING
   - Records all container reboots in /tmp/reboot-count.json
   - Tracks timestamps for past 1 hour
   - Alerts if >5 reboots detected in 1 hour window
   - Helps identify problematic containers

7. COMPREHENSIVE DOCUMENTATION
   - 6 markdown/txt documentation files
   - 2,000+ lines of guides, procedures, and references
   - Complete testing procedures with 10 test scenarios
   - Troubleshooting guides and examples

HOW IT WORKS - STEP BY STEP
================================================================================

1. USER REQUEST FLOW
   Step 1: User connects to http://localhost:4000 (nginx)
   Step 2: nginx applies round-robin load balancing
   Step 3: Request routed to app-4001 or app-4002
   Step 4: App instance processes request
   Step 5: Response returned through nginx to user

2. HEALTH CHECK MONITORING (Every 30 Seconds)
   Step 1: Docker executes health check command
   Step 2: Sends GET /api/health to app instance
   Step 3: App returns 200 (healthy) or 503 (shutting down)
   Step 4: If 200: Container marked healthy, traffic continues
   Step 5: If 503 or timeout: Failure count incremented
   Step 6: After 3 failures: Container marked unhealthy
   Step 7: nginx stops routing new requests to unhealthy container
   Step 8: Docker auto-restarts the container (restart: unless-stopped)

3. GRACEFUL SHUTDOWN FLOW
   Step 1: docker-compose stop/restart is issued
   Step 2: Container receives SIGTERM signal
   Step 3: process.on('SIGTERM') handler executes
   Step 4: isShuttingDown flag set to true
   Step 5: Reboot timestamp recorded in /tmp/reboot-count.json
   Step 6: Next health check returns 503
   Step 7: nginx detects failure, stops sending new requests
   Step 8: Existing requests complete (max 30 seconds)
   Step 9: Container exits cleanly
   Step 10: Docker auto-restarts container (unless-stopped)

4. REBOOT TRACKING FLOW
   Step 1: Container receives SIGTERM (shutdown initiated)
   Step 2: trackReboot() function executes
   Step 3: Read existing reboot timestamps from /tmp/reboot-count.json
   Step 4: Add current reboot timestamp
   Step 5: Remove timestamps older than 1 hour
   Step 6: If reboots.length > 5: Log warning message
   Step 7: Save updated file
   Step 8: Warning appears in Docker logs for ops review

TROUBLESHOOTING QUICK REFERENCE
================================================================================

ISSUE: Services won't start
  Step 1: Check logs: docker-compose logs
  Step 2: Verify env vars: docker-compose config
  Step 3: Check ports: lsof -i :4000 (if available)
  Step 4: Rebuild: docker-compose build --no-cache

ISSUE: Health checks failing
  Step 1: Check app logs: docker-compose logs app-4001
  Step 2: Check connectivity: docker-compose exec nginx ping app-4001
  Step 3: Test endpoint: curl http://localhost:4001/api/health
  Step 4: Restart: docker-compose restart app-4001

ISSUE: nginx can't reach backend
  Step 1: Verify app is running: docker-compose ps
  Step 2: Check network: docker network ls
  Step 3: Inspect logs: docker-compose logs nginx
  Step 4: Test directly: curl http://localhost:4001/api/health

ISSUE: High reboot count warning
  Step 1: Check file: docker-compose exec app-4001 cat /tmp/reboot-count.json
  Step 2: Check errors: docker-compose logs app-4001 | tail -50
  Step 3: Monitor resources: docker stats
  Step 4: Restart: docker-compose restart app-4001 app-4002

See HA_DEPLOYMENT.md for comprehensive troubleshooting guide.

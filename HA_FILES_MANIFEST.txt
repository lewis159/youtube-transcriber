ZERO-DOWNTIME UPDATE PROCEDURE
=============================

Scenario: Update app code and deploy without downtime

Steps:
  1. git pull origin main
  2. docker-compose build
  3. docker-compose restart app-4001
  4. sleep 10  # Wait for health check to pass
  5. docker-compose restart app-4002

Result:
  - app-4001 restarts while app-4002 serves all traffic
  - After app-4001 is healthy, app-4002 restarts
  - app-4001 handles traffic during app-4002 restart
  - No user-facing downtime


SCALING INSTRUCTIONS
===================

To add a 3rd instance (app-4003):

1. In docker-compose.yml:
   - Copy app-4002 section
   - Rename to app-4003
   - Change ports: 4003:3000

2. In nginx.conf:
   - Add to upstream: server app-4003:3000 max_fails=3 fail_timeout=30s;

3. Restart:
   docker-compose up -d app-4003
   docker-compose restart nginx

4. Verify:
   docker-compose ps  # Should show 5 services
   curl http://localhost:4003/api/health


TROUBLESHOOTING QUICK FIXES
===========================

Service won't start:
  → Check logs: docker-compose logs app-4001
  → Verify env vars: docker-compose config
  → Rebuild: docker-compose build --no-cache

Health checks failing:
  → Check app logs: docker-compose logs app-4001
  → Check connectivity: docker-compose exec nginx ping app-4001
  → Restart service: docker-compose restart app-4001

nginx can't reach backend:
  → Verify app is running: docker-compose ps
  → Check network: docker network ls
  → Inspect logs: docker-compose logs nginx

High reboot count:
  → Check file: docker-compose exec app-4001 cat /tmp/reboot-count.json
  → Check errors: docker-compose logs app-4001 | tail -50
  → Monitor resources: docker stats
  → Restart: docker-compose restart app-4001 app-4002

IMPLEMENTATION NOTES
====================

Decision Points:
  1. Grace Period: 30s chosen for typical video processing requests
  2. Health Check Interval: 30s chosen for responsive failure detection
  3. Load Balancing: Round-robin chosen for simplicity and even distribution
  4. Reboot Threshold: 5 in 1 hour chosen to catch problematic containers
  5. Two Instances: Minimum for HA (can scale to N instances)

Assumptions:
  - Docker and Docker Compose are installed
  - Environment variables are set before docker-compose up
  - Supabase and Clerk services are accessible
  - Redis is used for session storage
  - Users only access port 4000 (nginx)

Limitations:
  - Shared Redis (single point of failure for sessions)
  - Single Supabase project (external dependency)
  - Grace period may not be enough for very long operations
  - Reboot tracking is per-container (not cluster-wide)

Future Enhancements:
  - Add Redis replication/clustering
  - Add Supabase read replicas
  - Increase grace period dynamically based on request type
  - Implement distributed reboot tracking
  - Add automatic scaling based on CPU/memory metrics
  - Add persistent health check history
